## Pretrain
Train LLM to understand language, then tune it for end task
![[image1.png|500]]
**Objective function:Predict next token**
we use the **cross-entropy** loss to minimize the **deviation** between the distribution of the predicted tokens and the target token.
The resulting model is called **autoregressive**, or **Causal Language Model (CLM)**
**Data Mixture:**
● Web-scraped (e.g. Common Crawl, Wikipedia)
● Code (GitHub, StackOverflow)

==Notations:==
**FLOPs**: **FL**oating-point **OP**eration**s** 衡量计算量的基本单位
![[image2.png|500]]
可以理解为Function of **size of data(num of total tokens) and the num of parameters**
大部分时候取决于这两个因素 大致呈现 $O(\text{size of data} * \text{num of parameters})$ 

**FLOPS:**  **FL**oating-point **OP**erations per **S**econd (全部大写,实际上有时候会被混用)
How fast can your hardware execute these operations

------["Scaling Laws for Neural Language Models", Kaplan et al., 2020.]
Dataset越大,performance越好.

![[image4.png|550]]

## Training Optimization
Recap of LLM training:
**Initialization**:  **Model parameters**. O(billions) - O(100s of billions)scr
**Forward pass**   Compute loss:  你需要存储计算过程中的 **Activations** 来计算loss $\mathscr L(\theta_t)$   同时还有Function of model size, batch size, context length
**Backwards pass.** Compute gradients.  你需要存储**Gradients** for weights update.  $\nabla \mathscr L(\theta_t)$  
**Weights update.** Update model parameters.  你需要存储 Optimizer state. 我们通常用Adam optimizer $\theta_{t+1} \leftarrow \theta_t - \alpha \frac{m_t}{\sqrt{v_t} + \epsilon}$
一阶矩和二阶矩都需要存储(**梯度的moving average**和**梯度平方的moving average**)  即下面的数据
$m_{t+1} \leftarrow \beta_1 m_t + (1 - \beta_1)\nabla \mathcal{L}(\theta_t)$
$v_{t+1} \leftarrow \beta_2 v_t + (1 - \beta_2)(\nabla \mathcal{L}(\theta_t))^2$
我们需要存储如此多的内容 这就需要涉及**GPU的Memory** 而Bottleneck: memory
因此我们的思想就是把使用多块GPU能够让我们有更大的memory使用
### Data parallelism
Divide batch of data across devices
这就要求you have **a copy of the model per device**  因为你需要在每个设备计算activations做forward等等
能够有效**reduce the memory that is linked to the batch size.**
模型复制->数据划分（Sharding）(全局 batch = 4096  每块 GPU batch = 1024)
->并行计算前向 + 后向->梯度同步（AllReduce / NCCL）(设备之间做communications 然后**梯度平均（reduce）后，广播给所有 GPU** $g_i​→i=∑_{i}^N ​g_i​/N$)
问题在哪? 
1.不可避免地**communication cost**
2.存在大量的冗余信息
![[image8.png|600]]
在标准的 DP 中，**每张 GPU 都保存了完整的一份上述数据**  
既然大家都在做并行，为什么每张卡都要存一样的东西？切分它！

**Data parallelism with ZeRO**
["ZeRO: Memory Optimizations Toward Training Trillion Parameter Models", Rajbhandari et al., 2019.]
**以通信换显存**
Idea. Redundant information across devices
ZeRO = Zero Redundancy Optimization
每张 GPU 只维护 $\frac{1}{N}$ 的优化器状态,每张卡依然存有完整的 Parameters 和 Gradients，但只更新自己负责的那一小部分 Optimizer States
![[image9.png|400]]
每张卡在反向传播计算出梯度后，不再进行全量的 All-Reduce，而是只保留自己负责的那部分梯度，把其余的梯度发送给负责对应部分的 GPU
![[image10.png|400]]
**模型参数**本身也切分了

### Model Parallelism
大概了解一下
Split the model computations across several devices
**Tensor Parallelism（TP）**— 按“矩阵内部”切模型:
把一个大的矩阵（如 Attention/MLP 的权重）切成多块，让不同 GPU 各算一部分
**Pipeline Parallelism（PP）— 按“层”切模型**
把整个模型按层分到多张卡：  
GPU0 负责第 1~10 层，GPU1 负责 11~20 层……
**Sequence Parallelism（SP）— 按“sequence 维度”切输入序列**
把序列长度切分，例如：
- GPU0 处理 token 0–512
- GPU1 处理 token 513–1024
🟢 用于：长序列场景，attention 中间状态太大
**Context Parallelism（CP）— 扩展版的序列并行**
不仅把序列切开，还把 attention 计算按上下文分布式拆开，使非常长的上下文能分散到多 GPU
**Expert Parallelism（EP）— MoE 专家并行**
用于 MoE 模型。  
每个 Expert 放在不同 GPU，只在路由时激活少量 Experts


### Flash Attention(Important)
#### Standard Attention Implementation
我们这里做一个例子 
`batch_size = 1k`   `seq_length = 4k`  `hidden_dim = 4k`  `num_heads = 32`
+ Compute: The amount of computation required for softmax is a function of $4 × \text{batch size} × \text{seq length}^2 × \text{hidden dim}$. Given our assumptions this requires 256 TFLOPS($256×10^{12}$ FLOPS) of compute. 
+ Memory: Theamountofmemory required for the same softmax function is $\text{batch size}×\text{seq length}^2× \text{num heads}$. Given our assumptions this requires 512GB of memory.
The H100 SXM chip has 80GB of memory and has 989 TFLOPS of FP32 Tensor Core compute
所以我们想要用更多的computation换取我们去占用更少的memory

["FlashAttention: Fast and Memory-Efficient Exact Attention with IO-Awareness", Dao et al., 2022]
Leverage components of GPU to speed up attention computations
![[image12.png|500]]
HBM:高宽带内存  数十GB量级  传输速率每秒数TB级别
SRAM:静态随机存储器   几十MB量级  传输速率每秒数十TB级别
![[image13.png|500]]
设法利用这些memory的特性来**精确加速(不做任何approximation)**
leverage on cheap sram to prevent the cost of loading and writing large n byend attention matrics on slow hbm
BUT SRAM much smaller 所以我们一次性无法获得完整的attention矩阵

**Algorithm 0** Standard Attention Implementation
>**Require:** Matrices $\mathbf{Q}, \mathbf{K}, \mathbf{V} \in \mathbb{R}^{N \times d}$ in HBM.
	1: ==Load== $\mathbf{Q}, \mathbf{K}$ by blocks from HBM, compute $\mathbf{S} = \mathbf{Q}\mathbf{K}^{\text{T}}$, ==write== $\mathbf{S}$ to HBM.
	2: ==Read== $\mathbf{S}$ from HBM, compute $\mathbf{P} = \text{softmax}(\mathbf{S})$, ==write== $\mathbf{P}$ to HBM.
	3: ==Load== $\mathbf{P}$ and $\mathbf{V}$ by blocks from HBM, compute $\mathbf{O} = \mathbf{P}\mathbf{V}$, ==write== $\mathbf{O}$ to HBM.
	4: Return $\mathbf{O}$.

GPU的运算时间很快 但是大量时间却耗费loading your matrices from the memory
**can we compute the output without explicitly materializing S and P?**

#### Tiling 
因此我们想要做的就是Tiling操作
我们考虑AB两个矩阵进行相乘
![[image14.png|500]]
正常操作我们是去访问 **A的每一行的每一个值** 和 **B的每一列的每一个值** 然后计算出我们对应的C的元素
我们将矩阵划分为较小的**子矩阵（Block/Tile）** 计算过程从“行乘以列”转变为“小块乘以小块”
![[image15.png|240]]
![[image16.png|225]]
![[image17.png|225]]
逐渐转换成矩阵乘法
Memory Access Analysis
我们可以发现 without Tiling: 我们一共需要32 memory access
- 计算 $C_{1,1}$：需读 $A$ 第1行 (4个) + $B$ 第1列 (4个) = 8次
- 计算 $C_{1,2}$：需读 $A$ 第1行 (4个) + $B$ 第2列 (4个) = 8次
- 计算 $C_{2,1}$：需读 $A$ 第2行 (4个) + $B$ 第1列 (4个) = 8次
- 计算 $C_{2,2}$：需读 $A$ 第2行 (4个) + $B$ 第2列 (4个) = 8次
- **Total Accesses = $8 \times 4 = 32$ 次**
但是with tiling: 我们就只需要16memory access
将 $A$ 的相关行和 $B$ 的相关列一次性加载到片上内存
- **$A$ 的数据**：第1行和第2行，共 $2 \times 4 = 8$ 个元素。
- **$B$ 的数据**：第1列和第2列，共 $4 \times 2 = 8$ 个元素。
- **Total Accesses = $8 + 8 = 16$ 次**

实际上对于$N×N$的tile block 我们内存访问会变成$1/N$ 
#### Optimizing Softmax
我们知道softmax的计算公式: $y_i = \frac{e^{x_i}}{\sum_{k}^{V} e^{x_i}}$
但是存在一个比较大的问题:$e^{x_i}$ can result in large numbers, which open up the risk of numerical overflow leading to training instability
为例解决这个问题 我们用一个safe softmax的implementation来等价
$$\begin{align*}m_V = \max_{k=1}^{V} x_k \\
y_i = \frac{e^{x_i - m_V}}{\sum_{k}^{V} e^{x_k - m_V}}\end{align*}$$
本质上就是分子分母同时除以了最大值
>**Algorithm** Safe softmax
	1: $m_0 \leftarrow -\infty$
	2: $\text{for } k \leftarrow 1, V \text{ do}$
	3: $\quad m_k \leftarrow \max(m_{k-1}, x_k)$
	4: $\text{end for}$
	5: $d_0 \leftarrow 0$
	6: $\text{for } j \leftarrow 1, V \text{ do}$
	7: $\quad d_j \leftarrow d_{j-1} + e^{x_j - m_V}$
	8: $\text{end for}$
	9: $\text{for } i \leftarrow 1, V \text{ do}$
	10: $\quad y_i \leftarrow \frac{e^{x_i - m_V}}{d_V}$
	11: $\text{end for}$

我们现在依旧想要reduce global memory access 
我们试图想要把第一和第二个loop合在一起 但是有一个问题 有**Denpendency** 就是第二个loop需要$m_N$ $m_N$是整个sequence最大的值 
	我们之前的$d_i$表达式 : $d_i = \sum_{j=1}^{i} e^{x_j - m_N}$  
	我们想要$d_i$ 依赖于当前的最大值,而不是所有的最大值,这样就可以即读即算 :$d'_i = \sum_{j=1}^{i} e^{x_j - m_i}$
	我们只需要最后实现 $d'_N = d_N$ 这显然是成立
	我们将$d'_i$改写一下:  $d'_i = \left( \sum_{j=1}^{i-1} e^{x_j - m_i} \right) + e^{x_i - m_i}$
	然后做一些数学操作 $d'_i = \left( \sum_{j=1}^{i-1} e^{x_j - m_i} \right) e^{m_{i-1}} e^{-m_{i-1}} + e^{x_i - m_i}$
	我们做一个简单的swap $d'_i = \left( \sum_{j=1}^{i-1} e^{x_j - m_{i-1}} \right) e^{m_{i-1}} e^{-m_i} + e^{x_i - m_i}$
	我们最后就改写成了 $d'_i = d'_{i-1} e^{m_{i-1} - m_i} + e^{x_i - m_i}$

> **Algorithm:** Online softmax
	$m_0 = -\infty$ 
	$d_0 = 0$
	$\text{for } 1 \leq i \leq N \text{ do}$ 
		$m_i = \max(m_{i-1}, x_i)$
		$d'_i = d'_{i-1} e^{m_{i-1} - m_i} + e^{x_i - m_i}$
	$\text{for } 1 \leq i \leq N \text{ do}$ 
		$a_i = e^{x_i - m_N} / d'_N$


#### Flash Attention
**现在我们只考虑一行**
我们现在在softmax上面加上我们attention的计算
我们考虑一个query vector 
然后我们的$x_i$ 就等于我们这个query vector去dot product 所有的key vector 
然后我们就能算出我们的attention score 
那么我们的output就是attention score和value去做加权和
> Algorithm
	$m_0 = -\infty$
	$d_0 = 0$
	for $1 \leq i \leq N$ do
	  $x_i = qk_i^T$
	  $m_i = \max(m_{i-1}, x_i)$
	  $d'_i = d'_{i-1} e^{m_{i-1} - m_i} + e^{x_i - m_i}$
	$o_0 = 0$
	for $1 \leq i \leq N$ do
	  $a_i = e^{x_i - m_N} / d'_N$
	  $o_i = o_{i-1} + a_i v_i$
	return $o_N$

我们现在想要把这两个循环合成一个loop
依旧采用之前的办法
这是我们之前算法求output的表达式 $o_i = \sum_{j=1}^{i} \frac{e^{x_j - m_N}}{d'_N} v_j$
我们合并成一个loop 那么$m_N$和$d'_N$ 无法直接求得 那么我们还得要把改写成$m_i$和$d'_i$ : $o'_i = \sum_{j=1}^{i} \frac{e^{x_j - m_i}}{d'_i} v_j$ 
我们依旧是拆除掉第i项分开: $o'_i = \sum_{j=1}^{i-1} \frac{e^{x_j - m_i}}{d'_i} v_j + \frac{e^{x_i - m_i}}{d'_i} v_i$
然后做一个dummy term的代换 把1给拆开:  $o'_i = \sum_{j=1}^{i-1} \frac{e^{x_j - m_i}}{d'_i} \frac{d'_{i-1}}{d'_{i-1}} \frac{e^{-m_{i-1}}}{e^{-m_{i-1}}} v_j + \frac{e^{x_i - m_i}}{d'_i} v_i$
然后做swap $o'_i = \sum_{j=1}^{i-1} \frac{e^{x_j - m_{i-1}}}{d'_{i-1}} \frac{d'_{i-1}}{d'_i} e^{m_{i-1}-m_i} v_j + \frac{e^{x_i - m_i}}{d'_i} v_i$
我们可以发现前面实际上就是$o'_{i-1}$  $o'_i = \left( \sum_{j=1}^{i-1} \frac{e^{x_j - m_{i-1}}}{d'_{i-1}} v_j \right) \frac{d'_{i-1}}{d'_i} e^{m_{i-1}-m_i} + \frac{e^{x_i - m_i}}{d'_i} v_i$
那么我们最后就可以化成: $o'_i = o'_{i-1} \frac{d'_{i-1}}{d'_i} e^{m_{i-1}-m_i} + \frac{e^{x_i-m_i}}{d'_i} v_i$ 化成了recurrence的形式
所以我们最后算法就是
$$\begin{align*}
\textbf{for } 1 \leq i \leq N \textbf{ do} \\
x_i &= qk_i^{\text{T}} \\
m_i &= \max(m_{i-1}, x_i) \\
d'_i &= d'_{i-1}e^{m_{i-1}-m_i} + e^{x_i-m_i} \\
o'_i &= o'_{i-1} \frac{d'_{i-1}}{d'_i} e^{m_{i-1}-m_i} + \frac{e^{x_i-m_i}}{d'_i} v_i \\
\textbf{return } o'_N
\end{align*}$$

#### 算法讲解
$$
\begin{align*}
\textbf{Algorithm  }& \text{FLashAttention}\\
\textbf{Require: }& \text{Matrices } \mathbf{Q}, \mathbf{K}, \mathbf{V} \in \mathbb{R}^{N \times d} \text{ in HBM, on-chip SRAM of size } M.\\
\text{1: } & \text{Set block sizes } B_c \in \left[\frac{M}{4d}\right], B_r = \min\left(\left[\frac{M}{4d}\right], d\right). \\
\text{2: } & \text{Initialize } \mathbf{O} = \mathbf{0}_{N \times d} \in \mathbb{R}^{N \times d}, \ell = \mathbf{0}_{(\mathbf{N})} \in \mathbb{R}^N, m = (-\infty) \mathbf{N} \in \mathbb{R}^N \text{ in HBM}. \\
\text{3: } & \text{Divide } \mathbf{Q} \text{ into } T_r = \lceil \frac{N}{B_r} \rceil \text{ blocks } \mathbf{Q}_1, \dots, \mathbf{Q}_{T_r} \text{ of size } B_r \times d \text{ each, and divide } \mathbf{K}, \mathbf{V} \text{ into } T_c = \lceil \frac{N}{B_c} \rceil \text{ blocks } \\
& \quad \mathbf{K}_1, \dots, \mathbf{K}_{T_c} \text{ and } \mathbf{V}_1, \dots, \mathbf{V}_{T_c}, \text{ of size } B_c \times d \text{ each.} \\
\text{4: } & \text{Divide } \mathbf{O} \text{ into } T_r \text{ blocks } \mathbf{O}_1, \dots, \mathbf{O}_{T_r} \text{ of size } B_r \times d \text{ each, divide } \ell \text{ into } T_r \text{ blocks } \ell_1, \dots, \ell_{T_r} \text{ of size } B_r \text{ each, } \\
& \quad \text{divide } m \text{ into } T_r \text{ blocks } m_1, \dots, m_{T_r} \text{ of size } B_r \text{ each.} \\
\text{5: } & \textbf{for } 1 \leq j \leq T_c \textbf{ do} \\
\text{6: } & \quad \text{Load } \mathbf{K}_j, \mathbf{V}_j \text{ from HBM to on-chip SRAM.} \\
\text{7: } & \quad \textbf{for } 1 \leq i \leq T_r \textbf{ do} \\
\text{8: } & \quad \quad \text{Load } \mathbf{Q}_i, \mathbf{O}_i, \ell_i, m_i \text{ from HBM to on-chip SRAM.} \\
\text{9: } & \quad \quad \text{On chip, compute } \mathbf{S}_{ij} = \mathbf{Q}_i \mathbf{K}_j^{\text{T}} \in \mathbb{R}^{B_r \times B_c}. \\
\text{10: } & \quad \quad \text{On chip, compute } \mathbf{\tilde{m}}_{ij} = \operatorname{rowmax}(\mathbf{S}_{ij}) \in \mathbb{R}^{B_r}, \mathbf{\tilde{P}}_{ij} = \exp(\mathbf{S}_{ij} - \mathbf{1}\mathbf{\tilde{m}}_{ij}^{\text{T}}) \in \mathbb{R}^{B_r \times B_c} \text{ (pointwise)}, \\
& \quad \quad \mathbf{\tilde{\ell}}_{ij} = \operatorname{rowsum}(\mathbf{\tilde{P}}_{ij}) \in \mathbb{R}^{B_r}. \\
\text{11: } & \quad \quad \text{On chip, compute } m_i^{\text{new}} = \max(m_i, \mathbf{\tilde{m}}_{ij}) \in \mathbb{R}^{B_r}, \ell_i^{\text{new}} = e^{m_i-\mathbf{m}_i^{\text{new}}} \ell_i + e^{\mathbf{\tilde{m}}_{ij}-\mathbf{m}_i^{\text{new}}} \mathbf{\tilde{\ell}}_{ij} \in \mathbb{R}^{B_r}. \\
\text{12: } & \quad \quad \text{Write } \mathbf{O}_i \leftarrow \operatorname{diag}(\ell_i^{\text{new}})^{-1} (\operatorname{diag}(\ell_i)e^{m_i-\mathbf{m}_i^{\text{new}}} \mathbf{O}_i + e^{\mathbf{\tilde{m}}_{ij}-\mathbf{m}_i^{\text{new}}} \mathbf{\tilde{P}}_{ij} \mathbf{V}_j) \text{ to HBM}. \\
\text{13: } & \quad \quad \text{Write } \ell_i \leftarrow \ell_i^{\text{new}}, m_i \leftarrow m_i^{\text{new}} \text{ to HBM}. \\
\text{14: } & \quad \textbf{end for} \\
\text{15: } & \textbf{end for} \\
\text{16: } & \textbf{Return } \mathbf{O}.
\end{align*}
$$
我们一步一步来讲解一下原论文的思路
**准备:**
block size 通过计算公式算出 目的是为了把SRAM给打满
把结果矩阵$O$ (N×d) 初始化为0 后续递归累加 
类似的有$l$ (N) 对于每一行来说他的一个标量 就是指数和(我们上面的d) 一共有N行 所以是N维向量 也是初始化为0
$m$ (N) 对应每一行的最大值 初始化为-inf 

把 $\mathbf{Q} \in \mathbb{R}^{N \times d}$ 切分成 $T_r = \lceil \frac{N}{B_r} \rceil$ 个大小为 $B_r \times d$ 的块，把 $\mathbf{K}$ 和 $\mathbf{V}$ 切分成 $T_c = \lceil \frac{N}{B_c} \rceil$ 个大小为 $B_c \times d$ 的块。因此每次计算 $\mathbf{Q} \mathbf{K}^{\text{T}} \mathbf{V}$ 是 $B_r \times d$ 的 $\mathbf{Q}_i$ 和 $d \times B_c$ 的 $\mathbf{K}_j^{\text{T}}$ 和 $B_c \times d$ 的 $\mathbf{V}_j$，这样得到的最终大小是
$$(B_r \times d) \times (d \times B_c) \times (B_c \times d) = (B_r \times d).$$

所以我们的output矩阵也需要切分成$B_r\times d$的矩阵来存放中间结果
同样的长度为N的l和m也要切分成 $B_r$个元素的块 用来存放这些行的指数累加值和当前最大值

**算法执行**
![[image18.png|350]]

我们之前的思路是 针对每一个query 我们去计算于所有key的product 得到softmax 然后再去和value做加权 得到我们一行的值
这里其实非常诡异 FA1的两重循环中，是先外层循环load K, V，然后内层循环再load Q  这就会导致内层循环，每次计算的只是Qi的一部分，每次内循环的迭代都需要对Oi进行全局内存的读写 实际上**FA2就是KV是inner loop Q是outer loop 实际上这样也更符合我们的直觉** 也减少了一些读写
FA1 这个我们也可以理解 当考虑一行的时候我们是在求$m_i$ $l_i$ $o_i$ 然后来递归
我们现在实际上就是先一次求出当前k,v 去遍历q得到的小块block 然后拼接在一起 求出我们整块的$m_i$ $L_i$ $O_i$ $N×d$ 大小
然后遍历K,V的时候就能够实现更新

### Mixed Precision Training
#### Representation of a Float
![[image19.png|400]]
- **sign（1 bit）** —— 决定正负
- **exponent（指数字段）** —— 决定范围（能表示多大、多小）
- **mantissa / significand（尾数字段）** —— 决定精度（能表示小数多精细）
$$
\text{value}=(−1)^s⋅2^{e−bias}⋅(1.f)
$$
其中 $1.f=1+\frac{b_1}{2^1}​​+\frac{b_2}{2^2}​​+⋯+\frac{b_t}{2^t}​​$ 
	t = mantissa bits（FP32=23，FP16=10，BF16=7）

| 格式       | 总比特数 | 指数比特  | 尾数比特  | 精度  | 可表示范围        |
| -------- | ---- | ----- | ----- | --- | ------------ |
| **FP32** | 32   | 8     | 23    | 高   | 极大/极小范围      |
| **FP16** | 16   | 5     | 10    | 中   | 范围较小         |
| **BF16** | 16   | **8** | 7     | 低   | 范围与 FP32 一样大 |
| **FP8**  | 8    | 4 或 5 | 3 或 2 | 很低  | 范围更小         |
- 梯度太小 → underflow（变成 0）
- 梯度太大 → overflow（变成 inf）
- 表示不精确 → loss 不下降
- Tensor Core 要求特定精度 → 速度差异巨大 **Lower precision → Faster processing**
#### Mixed Precision Training
既保持 FP32 的训练稳定性，又利用 FP16/BF16 的计算速度
+ 模型参数保持 FP32（主权重）
+ 前向推理计算用 FP16 / BF16
	矩阵乘法 在 Tensor Core 上跑，速度极快 **autocast** 自动做的
+ 梯度也通常以 FP16/BF16 计算，但在更新前 cast 回 FP32
		loss.backward() 得到 FP16/BF16 梯度
		半精度梯度被 **cast 回 FP32**
		用 FP32 版本的 optimizer（如 AdamW）更新权重
		更新的 FP32 权重再 cast 成 FP16/BF16 用于下一次 forward

## Supervised finetuning
![[image21.png|500]]
SFT的底层目标: **让模型把“用户输入”理解成“指令”，并产生“任务型输出”**
pretrain 之后的大模型“能生成任何东西”，但不能“按你的任务来”
SFT 强制模型把“任务 → 输出”的 mapping 固定下来
✔ 改变模型的“解码偏好” 它改变的是模型的“行为生成策略”，不是“知识”
**一些训练策略:**
⭐ Strategy 1：数据格式的一致性 > 数据内容质量
⭐ Strategy 2：数据要“任务化”，不是“聊天化”
不好数据（bad）：
`User: 什么是黑洞？ Assistant: 黑洞是……`
优秀数据（good）：
`Instruction: 请用高中物理水平解释黑洞的形成原因。 Answer: 好的，我将从三个方面解释黑洞……`
⭐ Strategy 3：SFT 学习率不能大
典型 LR：
- 7B：1e–5
- 13B：5e–6   
- 34B：2e–6  
- 70B：1e–6 或更小
⭐ Strategy 4：SFT 步数不要太长（守住能力上限）

## Lora  
**出发点不是“减少参数”，而是两个核心问题**
（1）大模型全量微调会破坏模型能力: 全量 SFT = 修改所有 W 但大模型内部表示非常脆弱（尤其是高层 FFN）
（2）需要“添加”能力，而不是“覆盖”能力 **不需要修改整个表示空间，只需要添加一些方向/偏置** 

 🧠  LoRA 到底作用在哪里？
👉 作用在 **attention** 和 **MLP** 的线性层：
通常选：
- q_proj（最常选）
- v_proj（强烈影响输出风格）
- o_proj（影响 token mixing）
- k_proj（改变 key 空间，不稳定）
- gate_proj / up_proj / down_proj（影响表达能力，非常重要）  

**实战经验：**
- **只加 q_proj → 模型“关注点”变了**
- **再加 v_proj → 模型输出风格变明显**
- **加到 MLP → 模型能力变强/更具体（适合任务）**
很多人误以为 attention 就够了，  
但==**加 MLP（FFN）才是大任务里效果的关键!!!!**== 

+ 一般模型越大 需要的Lora Rank越小
+ LoRA dropout 不设，会导致过拟合 
+ 用太大的 rank 会降低训练速度  r=64 时开销可能比全量还大
+ LoRA needs a higher learning rate than full fine-tuning
+ 🔥 **LoRA 在大 batch 下比较容易不稳定、训练发散或训练质量差** FFT:  更新 1000 层，每层噪声互相抵消

### QLora
Quantization: 将深度学习模型中原本使用 32 位浮点数（FP32）或 16 位浮点数（FP16/BF16）表示的参数（Weights）和激活值（Activations），映射到 **8 位整数（INT8）** 或 **4 位整数（INT4）** 的过程
$$q = \text{clip}\left( \text{round}\left( \frac{r}{S} + Z \right), q_{\min}, q_{\max} \right)$$
**$S$ (Scale, 缩放因子)**: 决定了量化的步长，通常是 $S = \frac{r_{\max} - r_{\min}}{q_{\max} - q_{\min}}$
**$Z$ (Zero-point, 零点)**: 浮点数中的 $0$ 对应整数中的哪个值 这主要用于处理非对称分布（例如 ReLU 后的数据全是正数）
**$q_{\min}, q_{\max}$**: 整数的范围。
- **INT8**: $[-128, 127]$ (有符号) 或 $[0, 255]$ (无符号)
- **INT4**: $[-8, 7]$ (有符号) 或 $[0, 15]$ (无符号)
INT8 量化 (8-bit)  256 个离散值
**W8A8 (Weight 8-bit, Activation 8-bit)**: LLM 的激活值存在 **Outliers (离群值)** 现象（某些通道数值特别大），直接做 INT8 会导致精度崩塌
INT4 量化 (4-bit)
**W4A16 (Weight 4-bit, Activation 16-bit)** 不是为了算得更快（很多硬件不支持原生的 INT4 计算），而是为了**减少显存占用**和**减少显存带宽压力**


1,Motivation: 让我们不牺牲性能的前提下，用极低显存微调超大模型
2,Problem: 
❗ LoRA 虽然减少了**训练参数数量** 没有减少主模型的显存，只减少训练的参数 你一个33B的model 用FP16 都要66GB 单卡80GB你都跑不动
❗QLoRA 之前的 INT8/INT4 量化不够稳定 这玩意量化误差太大了    loss 发散,梯度震荡,下游任务难以收敛,模型精度下降明显  训不动
3.Method:
**把模型主权重压缩到 4bit（NF4/FP4），但训练时用 LoRA 在高精度空间加一个增量。**  
**最终的 forward = dequantize(W_4bit) + LoRAΔW**
🧩 3.1 NF4：为 LLM 权重专门设计的 4bit 量化格式
LLM权重在统计上近似Guassian 而非均匀分布 INT4等长分布固然不合适
将高斯分布的 CDF 分成 16 份（对应 4bit） 每一个编码值对应一个高斯分布的分位点
![[image22.png|500]]
Experiment:🟩 NF4 量化后的模型性能非常接近 FP16
🧩 3.2 Double Quantization（双重量化）
每个矩阵你还需要乘以一个 scale factor 
如果每行 / 每列都需要存 scale → overhead 很大
Qlora 直接把Scale 也做一次8bit量化 
最终每个矩阵几乎没有额外显存开销
🧩 3.3 LoRA in FP16/BF16（增量保持高精度）
主模型权重 W：
- 存储为 4bit
- 不更新
LoRA 权重 ΔW：
- 存储为 FP16/BF16
- 使用高精度训练
- 增量大小很小
Forward时候: W_eff = dequant(W_4bit) + BA
Backward 时：W_4bit 不更新  BA 更新
🧩 3.4 Paged Optimizer（分页优化器）
⭐ 使用类似虚拟内存的“分页”机制，把 optimizer 状态放入 CPU 内存
- GPU 只存 LoRA 权重和激活
- 所有 Adam 状态放到 CPU RAM
- 再也不用 GPU 背负巨量 optimizer 显存
4 .QLoRA 训练流程
- **加载模型参数（4bit NF4）**
- **把 scale（8bit）解码**
- **forward 时 dequantize 成 FP16 进行计算**
- **计算 QKᵀ、KV 等 attention 逻辑**
- **加上 LoRA 的 BA 矩阵（FP16）**
- **计算 loss**
- **backward 时只更新 BA**
- **Adam 状态放在 CPU（paged optimizer**

QLoRA 把主模型放进一个极低精度的 4bit 空间来节省显存，但用 LoRA 在高精度空间训练增量，从而实现“低显存 + 高精度微调”的完美组合

## DoRA
Weight-Decomposed Low-Rank Adaptation
DoRA = 改进版 LoRA：把权重分成 “方向（direction）+ 尺度（magnitude）
**只对方向做低秩微调，而不破坏尺度结构  结果：更稳定、更强、更抗大 batch**
1.Motivation
直接在 W 上加一个低秩扰动（BA），会破坏 W 的范数结构
Transformer 中**权重的范数（norm）结构非常敏感**  （是稳定性、梯度、激活分布的关键）
2,Method
$$W = m \frac{V}{||V||_c}$$
- $||V||_c$ 表示对 $V$ 的每一列（或行，取决于定义，通常是输出维度）求范数
- $\frac{V}{||V||_c}$ 保证了方向矩阵是单位向量
![[image23.png|700]]
**对于 Direction ($V$)**：保持冻结，使用 **LoRA** 来进行微调。即 $V' = V_0 + \Delta V = W_0 + BA$
**对于 Magnitude ($m$)**：作为一个可训练的**向量**，直接进行微调（全量微调 $m$，因为 $m$ 参数量极小，只有 $d_{out}$ 个参数）
优点 (Pros)
1. **性能更强**：在 LLaMA、VL 等多个模型上，DoRA 的准确率普遍高于 LoRA，甚至在某些任务上超过了全量微调
2. **学习动力学更优**：分析表明 DoRA 的更新模式确实更接近 Full FT
3. **鲁棒性**：对于秩 $r$ 的选择不那么敏感，通常比 LoRA 更稳
4. **推理无负担**：合并后与 Base Model 速度一致
缺点 (Cons)
5. **训练显存略增**：需要额外存储 $m$ 向量（极小，可忽略）以及计算图中的归一化操作产生的中间激活值（这部分比 LoRA 略大）
6. **训练速度略慢**：因为多了 Norm 计算和反向传播中的除法链式法则，比标准 LoRA 慢一点点（通常 <20%）
7. **QDoRA 实现复杂**：如果你想把 DoRA 和 QLoRA (4-bit) 结合（即 **QDoRA**），实现起来比较麻烦，因为需要对量化的权重进行归一化，这对精度控制要求很高（不过 Unsloth 等库已经解决了这个问题）

### AdaLora
1.Motivation: 
Transformer 层内部的重要性差很大：
- 有些多头 attention heads 对任务至关重要
- 有些 value projection 几乎无影响
- 有些 FFN 层极度关键
- 有些层任务几乎不用到（尤其深层/浅层）
❗ **LoRA 给每层固定相同的 rank = 在浪费参数**
- 不重要的层分到了太多参数
- 重要的层分到的 rank 太少（造成性能瓶颈）
→ **表达能力不够 + 参数浪费**
2.Method
**使用训练过程中产生的统计量（如梯度重要性、奇异值等），给每一层分配动态 rank**
Step 1 → Warmup：所有层使用大 rank（例如 r=64）
	每层用一个较大 rank（r=64 或 r=32）,  短时间训练（几千 steps）
Step 2 → Rank Allocation：计算每层“重要性分数”，并降低不重要层的 rank
	Pruning AdaLoRA 将其建模为 SVD 的形式 $$\Delta W = P \Lambda Q$$
	Magnitude of Gradient-Weight Product（梯度-权重乘积的幅值）。
	对于第 $k$ 个三元组 $(P_k, \lambda_k, Q_k)$，其重要性得分 $S_k$ 定义为：$$S_k = s(\lambda_k) + \text{Mean}(s(P_k)) + \text{Mean}(s(Q_k))$$其中 $s(\cdot) = |w \cdot \nabla w|$
    **直觉**：如果改变某个参数会导致 Loss 剧烈变化（梯度大），且该参数本身数值也不小，那么这个参数就非常重要
    计算所有层中所有奇异值的重要性得分
	将得分排序
    **Mask** 那些得分最低的奇异值，直到满足当前的参数预算
    被置零的 $\lambda_k$ 对应的 rank 就实际上消失了
Step 3 → Finetune：用稀疏的 LoRA（不同层不同 rank）继续训练
